How an Arduino Sketch is Structured

Every Arduino sketch has the same basic shape. Understanding this shape helps you read any sketch, not just this one.

// 1. Library includes
#include "esp_camera.h"
Libraries are collections of pre-written code. #include loads them into your sketch so you can use their functions without writing everything yourself. esp_camera.h is the camera library — it knows how to talk to the OV2640 camera chip.
// 2. Constant definitions
#define CAR_NAME "SpyCar_01"
#define WATCHDOG_MS 1500
#define creates a named constant — a label that is replaced by its value everywhere in the code before compiling. CAR_NAME will be replaced by "SpyCar_01" wherever it appears. This is easier to change than finding every occurrence of the text.
// 3. Global variables
WebServer pageServer(80);
unsigned long lastCmdTime = 0;
Global variables are created outside any function and exist for the entire life of the program. pageServer is the web server object that handles port 80. unsigned long is a data type — a positive whole number up to ~4.3 billion (large enough to store milliseconds since boot).
void motorForward() { ... }
void motorLeft() { ... }
Functions are named blocks of reusable code. void means the function doesn't return a value — it just does its job. The name followed by () is how you call (run) it from elsewhere. Parentheses can contain parameters, but these motor functions need none.
void setup() { ... }
void loop() { ... }
setup() is Arduino's special startup function — it runs exactly once when the board is reset. loop() is Arduino's main function — it runs over and over, as fast as possible, forever. All Arduino sketches must have these two functions.

Technical Vocabulary — Glossary

void
Return type keyword

Means "returns nothing". When a function's job is to DO something (like set a pin HIGH) rather than CALCULATE something, its return type is void. e.g. void motorForward() just drives the motors — it doesn't need to give back a number.

setup() / loop()
Special Arduino functions

setup() runs once at boot — use it to initialise hardware. loop() runs repeatedly forever — use it to check for inputs and respond. Think of setup as "getting the kitchen ready" and loop as "cooking continuously".

#define
Preprocessor directive

Creates a named constant that is substituted before compiling. #define WATCHDOG_MS 1500 means wherever WATCHDOG_MS appears in the code, the compiler replaces it with 1500. Easier to change in one place than hunting through code.

static const int
Variable declaration

static = only visible in this file. const = cannot change after declaration. int = whole number data type. Together: a file-local constant integer. We use this for GPIO pin numbers because they never change and don't need to be visible outside this file.

digitalWrite(pin, value)
Built-in Arduino function

Sets a GPIO pin to HIGH (3.3V) or LOW (0V). We use this to control the L298N H-bridge inputs. digitalWrite(IN1, HIGH) sets GPIO13 to 3.3V, which tells the L298N to drive current one way through the left motor.

pinMode(pin, mode)
Built-in Arduino function

Configures a GPIO pin as OUTPUT (we control it) or INPUT (we read it). We call pinMode(IN1, OUTPUT) in setupMotors() so the ESP32 knows we want to drive signals out on those pins, not read signals in.

millis()
Built-in Arduino function

Returns the number of milliseconds since the board was last reset or powered on. Used in the watchdog: millis() - lastCmdTime > 1500 checks if 1.5 seconds have passed without a motor command. Wraps around (resets to 0) after ~49 days.

WebServer / httpd
HTTP server objects

WebServer (port 80) is the Arduino-style server — simple to use, handles the control page and motor commands. httpd_handle_t (port 81) is the ESP-IDF low-level server — faster for streaming. Using two separate servers is FIX 2 — they never block each other.

PROGMEM
Memory qualifier

Stores data in flash (program) memory instead of RAM. The HTML page (const char PAGE[] PROGMEM) is ~2 KB. Storing it in PROGMEM frees that RAM for the camera and WiFi stack. send_P() reads directly from PROGMEM when sending to browser.

MJPEG
Streaming video format

Motion JPEG — a stream of individual JPEG images sent over HTTP, one after another. The browser displays them in sequence to create video. Simpler than H.264 and well-supported. Each frame is separated by a boundary marker string so the browser knows when one image ends and the next begins.

PSRAM
External RAM chip

Pseudo-Static RAM — an extra 4 MB RAM chip soldered next to the ESP32 on the AI Thinker board. Without it, there is only ~320 KB internal RAM — too small for large camera frames. With PSRAM, VGA (640×480) is possible. The sketch checks psramFound() and falls back to QVGA if absent.

GPIO
Hardware term

General Purpose Input/Output — the numbered pins on the ESP32-CAM that can be programmed to read or write digital signals. GPIO13, 14, 15, 12 control the motor H-bridge. GPIO4 controls the flash LED. GPIO16 is reserved for PSRAM and must NOT be used as a motor pin.

The Motor Direction Bug — Why It Happened and How v10 Fixes It

🐛 The Bug in v9 In v9, pressing Forward made the car turn left/right, and pressing Left/Right made the car go straight forward or backward. The code logic was not wrong — the physical motor mounting was not accounted for.

Why does mounting direction matter?

The two TT motors are mounted facing each other inside the chassis — a mirror image. Because they face opposite directions, the same electrical signal produces opposite physical rotation for each motor.

Imagine you and a friend stand face-to-face and both spin your right arms forward. Your arm goes one way, their arm goes the other — even though you did the same action. This is what happens with mirrored motors.

FunctionIN1IN2IN3IN4v9 Physical Resultv10 Physical Result
motorForward()HLHLPivots left ❌Drives forward ✅
motorBackward()LHLHPivots right ❌Drives backward ✅
motorLeft()LHHLDrives forward ❌Pivots left ✅
motorRight()HLLHDrives backward ❌Pivots right ✅

The v10 fix accounts for the mirrored mounting. To drive FORWARD (both wheels physically forward), the two motors need opposite electrical signals:

// v10 motorForward() — corrected for mirrored motor mounting
void motorForward() {
  // Left  motor: IN1=LOW, IN2=HIGH  → left  wheel turns FORWARD physically
  digitalWrite(IN1, LOW);  digitalWrite(IN2, HIGH);
  // Right motor: IN3=HIGH, IN4=LOW → right wheel turns FORWARD physically
  // (opposite signal because the right motor is physically reversed)
  digitalWrite(IN3, HIGH); digitalWrite(IN4, LOW);
}

setup() and loop() — The Heart of Every Arduino Sketch

void setup() {
  // 1. Disable brownout detector ─────────────────────────────────────────────
  //    Motors draw a current spike on startup that momentarily drops the
  //    supply voltage. Without this line, the board would keep resetting.
  WRITE_PERI_REG(RTC_CNTL_BROWN_OUT_REG, 0);

  // 2. Start serial communication ────────────────────────────────────────────
  //    115200 baud = 115,200 bits per second. This lets us print debug
  //    messages to Arduino IDE's Serial Monitor.
  Serial.begin(115200);

  // 3. Initialise motors, LED, camera ────────────────────────────────────────
  setupMotors();                            // Set pins as OUTPUT, stop motors
  pinMode(FLASH_LED_PIN, OUTPUT);           // LED pin as output
  digitalWrite(FLASH_LED_PIN, LOW);        // LED off at startup
  setupCamera();                            // Configure and start camera

  // 4. Start WiFi (mode determined by #define at top of file) ───────────────
  WiFi.mode(WIFI_AP);                        // AP = Access Point (hotspot)
  WiFi.softAP(AP_SSID, AP_PASS);           // Create the hotspot network

  // 5. Register URL routes on port 80 ────────────────────────────────────────
  pageServer.on("/", []() { ... });       // Serve the HTML control page
  pageServer.on("/cmd", []() { ... });   // Handle motor commands
  pageServer.on("/flash", []() { ... }); // Handle LED commands
  pageServer.begin();

  // 6. Start stream server on port 81 ────────────────────────────────────────
  startStreamServer();
}
void loop() {
  // Process any waiting HTTP requests (button presses from browser).
  // Without this line the buttons would never work!
  pageServer.handleClient();

  // ── MOTOR WATCHDOG ──────────────────────────────────────────────────────
  // millis() = milliseconds since boot.
  // lastCmdTime was set when the last /cmd arrived.
  // If the gap exceeds WATCHDOG_MS (1500 ms) → stop motors.
  // Safety: if WiFi drops mid-drive, car stops automatically.
  if (lastCmdTime > 0 && millis() - lastCmdTime > WATCHDOG_MS) {
    motorStop();
    lastCmdTime = 0;   // Reset so watchdog doesn't keep firing
  }

  delay(5);  // Prevent CPU from running at 100% unnecessarily
}

The MJPEG Stream — FIX 2 Explained

⚠️ The problem this solves (v8 bug) In earlier versions, the stream and motor commands ran on the same HTTP server. When the camera was sending a frame (which can take 30–100 ms on slow WiFi), any button press was queued waiting for the frame to finish. The car felt unresponsive or froze mid-turn. FIX 2 separates them onto different ports. 
// Port 80 — WebServer (Arduino library)
// Handles: HTML page, /cmd?go=X, /flash?val=X
// Fast responses, never blocked by the stream.
WebServer pageServer(80);

// Port 81 — esp_http_server (ESP-IDF lower-level library)
// Handles: /stream only — loops sending JPEG frames forever.
// Runs independently; cannot block the controls on port 80.
httpd_handle_t streamServer = NULL;

// Inside streamHandler() — the frame-sending loop:
while (true) {
  camera_fb_t* fb = esp_camera_fb_get();  // Wait for next JPEG frame
  // ... send boundary marker ...
  // ... send JPEG header (size) ...
  // ... send JPEG image data ...
  esp_camera_fb_return(fb);   // CRITICAL: return buffer to camera
  delay(STREAM_DELAY);        // 30 ms between frames
}

Network Plan

20 Cars Simultaneously — How It Works

In Access Point (hotspot) mode, each car creates its own isolated WiFi network. There are no IP address conflicts even with 20 cars in the same room.

✅ The clever part: each car IS its own router In AP mode every ESP32-CAM assigns itself the address 192.168.4.1 and runs its own private network (e.g. SpyCar_01). These networks don't interfere because they are isolated — each student's phone or laptop connects to only one of them. It's like 20 separate rooms, each with the same house number, on completely different streets.

Option A — Standalone Hotspot (Default)

✅ Recommended for class use

No school network or router needed. Works anywhere. Students connect phone/laptop to their car's WiFi hotspot. URL is always 192.168.4.1. 20 cars = 20 separate networks — zero interference.

⚠️ Internet access is off while connected to car hotspot This is normal and expected. Students switch back to school WiFi when they need the internet (e.g. for OneNote). Remind them to switch back at the end of the lesson.

Option B — School Network (Backup)

⚙️ When to use this

Use if the standalone hotspot causes connectivity issues. The car joins the school WiFi. IP address is dynamic — read it from Serial Monitor. All devices must be on the same network. To activate: uncomment #define WIFI_MODE_STA in the sketch.

Note for IT administrator If using Station mode, the school DHCP server assigns IPs. For stability, assign fixed/reserved IPs based on each ESP32-CAM's MAC address. MAC address is printed in Serial Monitor on first boot.

General Practical Lesson Safety — Non-Negotiable Rules

👁️

Active Supervision at All Times

You must be physically present and observant in the room for the entire lesson. You cannot sit at the teacher's desk and mark rolls while students use tools or electrical components. Circulate every 3–5 minutes. Maintain line-of-sight with all students.

🦺

PPE and Safety Equipment

Enclosed footwear is mandatory (school uniform policy). Safety glasses are available in the labelled cabinet — issue them before any cutting activity. Students must remove dangling jewellery and tie back long hair before using any tools or powering on electrical components.

🔋

Power-On Approval Required

No student may connect battery power to their car without teacher inspection. Walk to each group, visually inspect the wiring against the provided wiring diagram, and verbally approve before they turn on the Pololu-1771 switch. Initial their safety checklist in the logbook.

🚗

Driving Tests — Floor Only

All car driving tests must be conducted on the classroom floor in the designated test zone. Cars must not be driven on desks, benches, or chairs. Clear the test zone of bags, chairs, and cables before any driving test. Students must stand back when the car is moving.

📋

Seating and Roll

Direct students to their allocated seats (seating chart on board). Year 9 students must not sit adjacent. Mark roll in eSIS within first 10 minutes. Record any student not in their allocated seat.

📦

Equipment Storage

All electronic components (ESP32-CAM boards, L298N modules, batteries) must be returned to their labelled storage boxes at the end of each lesson. Count components before and after. Report any missing items to the HoD before leaving.
